# Required MobiFlight Core Modifications

The CC_G5 firmware requires three changes to the MobiFlight core source files
in `src/src/`. These files are downloaded by the build system — make the changes
after running the initial build (or after `get_CoreFiles.py` runs) and before
flashing.

---

## 1. `src/src/MF_Button/MFButton.cpp` — Disable physical button setup

The CC_G5 uses virtual buttons (triggered by firmware events, not GPIO pins).
The core's `MFButton::attach()` must not call `pinMode()` or `digitalRead()` on
these virtual pin numbers.

**Find** the `MFButton::attach()` function:

```cpp
void MFButton::attach(uint8_t pin, const char *name)
{
    _pin  = pin;
```

**Add `return;` as the very first line of the function body:**

```cpp
void MFButton::attach(uint8_t pin, const char *name)
{
    return;   // CC_G5: skip physical pin setup — all buttons are virtual
    _pin  = pin;
    _name = name;
    pinMode(_pin, INPUT_PULLUP);
    _state       = digitalRead(_pin);
    _initialized = true;
}
```

> **Why:** The Connector config includes button records with arbitrary dummy pin
> numbers so users can map events to them. Without this change the firmware would
> call `pinMode()` on those dummy pins at startup, causing crashes or incorrect
> GPIO behaviour on the ESP32-S3.

## 2. `src/src/MF_Encoder/Encoder.cpp` — Disable physical encoder setup

The CC_G5 uses virtual encoders driven by I2C from the RP2040 co-processor, not
direct GPIO. The core's `Encoder::Add()` must not attempt to register or
configure physical pins for these.

**Find** the `Encoder::Add()` function inside `namespace Encoder`:

```cpp

{

```

**Add `return;` as the very first line of the function body:**

```cpp

{
    return; // CC_G5: skip physical pin setup — all encoders are virtual (I2C via RP2040)
    if (encodersRegistered == maxEncoders)
        return;
    encoders[encodersRegistered] = MFEncoder();
    encoders[encodersRegistered].attach(pin1, pin2, encoder_type, name);
    MFEncoder::attachHandler(handlerOnEncoder);
    encodersRegistered++;

```

> **Why:** Encoder events are generated by the firmware itself based on I2C
> messages from the RP2040. The MobiFlight encoder polling loop must not run on
> these dummy pin numbers. The encoder/button names still appear in the Connector
> (via the virtual config injection below) so users can map them to sim variables.

## 3. `src/src/Config.cpp` — Inject virtual device records into `kGetConfig` response

When the Connector sends command `12;` to read the board config, the firmware
must append virtual encoder and button records that match the selected device
type (HSI, PFD, or switchable). These records are not stored in EEPROM — they
live in firmware flash in `MFCustomDevicesConfig.h` and are injected at response
time.

### 3a. Add the forward declaration

**Find** the `HAS_CONFIG_IN_FLASH` include block (near the top of `Config.cpp`):

```cpp
#ifdef HAS_CONFIG_IN_FLASH
#include "MFCustomDevicesConfig.h"
#else
const char CustomDeviceConfig[] PROGMEM = {};
#endif
```

**Replace it with:**

```cpp
#ifdef HAS_CONFIG_IN_FLASH
#include "MFCustomDevicesConfig.h"
// Implemented in MFCustomDevice.cpp — reads the device type from EEPROM at
// typeAddrEEPROM and streams the matching virtual encoder/button records into
// the kGetConfig response so the Connector can see and map to them.
void appendVirtualDeviceConfig(uint16_t typeAddrEEPROM);
#else
const char CustomDeviceConfig[] PROGMEM = {};
inline void appendVirtualDeviceConfig(uint16_t) {}
#endif
```

### 3b. Call the function in `OnGetConfig()`

**Find** the `OnGetConfig()` function and its EEPROM streaming block:

```cpp
    if (configStoredInEEPROM()) {
        cmdMessenger.sendCmdArg((char)MFeeprom.read_byte(MEM_OFFSET_CONFIG));
        for (uint16_t i = 1; i < configLengthEEPROM; i++) {
            cmdMessenger.sendArg((char)MFeeprom.read_byte(MEM_OFFSET_CONFIG + i));
        }
    } else if (configStoredInFlash()) {
```

**Add the call immediately after the closing brace of the EEPROM loop:**

```cpp
    if (configStoredInEEPROM()) {
        cmdMessenger.sendCmdArg((char)MFeeprom.read_byte(MEM_OFFSET_CONFIG));
        for (uint16_t i = 1; i < configLengthEEPROM; i++) {
            cmdMessenger.sendArg((char)MFeeprom.read_byte(MEM_OFFSET_CONFIG + i));
        }
        // Append virtual encoder/button records from firmware flash, based on device type.
        // These devices have no physical pins but must appear in the Connector config
        // so users can map sim variables to them.
        // The type string in EEPROM starts at MEM_OFFSET_CONFIG + 3 (after "17.").
        appendVirtualDeviceConfig(MEM_OFFSET_CONFIG + 3);
    } else if (configStoredInFlash()) {
```

> **Why:** The user's device type (e.g. `CC_G5_HSI`) is stored in EEPROM by the
> Connector. The matching virtual encoder/button records (e.g. `encHeading`,
> `encCourse`, `btnHsiEncoder`) are defined in firmware flash in
> `MFCustomDevicesConfig.h`. Injecting them here means the Connector receives a
> complete config and can present the full list of mappable inputs to the user,
> without requiring the user to manually add encoders and buttons in the UI.
